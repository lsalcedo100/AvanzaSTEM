# Avanza STEM
#### Video Demo: https://www.youtube.com/watch?v=eyTIcIG6QnU
#### Description:

Avanza STEM is a multilingual educational website built to make science, technology, engineering, and math feel inviting, creative, and achievable for young learners. The audience I had in mind while building it was elementary and middle-school students, especially Hispanic students who may not always see enough free, community-centered STEM resources designed with them in mind. Instead of making the site feel like a formal school portal, I wanted it to feel energetic and welcoming, with strong colors, big visuals, simple navigation, and content that explains STEM ideas in a hands-on way. The result is a website that acts as a small learning hub: it introduces the mission of the organization, showcases possible curriculum tracks, offers project guides, highlights blog-style learning content, and advertises in-person workshops.

From a user perspective, the website is organized into five major sections. The landing page introduces the mission and shows featured activities, statistics, and calls to action. The curriculums page presents several learning tracks such as Python, engineering, science, math, robotics, and AI. The projects page is the most hands-on area of the app, because it lets students browse DIY activities by category and then open a full guide. The blog page is designed to feel like a library of approachable STEM stories and tutorials, while the workshops page makes the in-person part of the organization feel real by showing upcoming events, past workshops, and community outreach information. Together, those sections make the project feel less like a single webpage and more like a complete digital home for an after-school STEM initiative.

Technically, the project is built with Next.js, React, TypeScript, and Tailwind CSS. I used the App Router structure because it makes the site easy to organize by route, and it matches the way a user thinks about the platform: each major experience lives at its own page path. Tailwind made it practical to iterate quickly on cards, responsive layouts, gradients, and spacing without needing a huge amount of custom CSS. The app also uses Vercel Analytics, which gives the project a more realistic deployment setup and makes it feel closer to a production-ready site rather than just a static mockup.

One of the biggest structural decisions I made was to separate content from presentation wherever possible. The file `features/projects/data.ts` is the best example of that choice. Instead of hardcoding project details directly into every page, I created a central data source for project metadata and guide content such as titles, difficulty labels, materials, steps, safety notes, and challenge prompts. That makes the project easier to maintain because I can update one file and have the changes flow into the projects listing and the guide pages. I took the same approach with `i18n/translations.ts`, where I grouped interface text and page copy by language. Right now the site supports English, Spanish, and Chinese, which reflects another core goal of the project: accessibility. The language system is not just decorative; it changes the text users see throughout the site and makes the platform feel more inclusive.

The shared application shell is handled by a small group of important files. `app/layout.tsx` defines global metadata, loads the site font, wraps the site in the language provider, and renders the navbar and footer around every page. `components/providers/language-provider.tsx` manages the active language on the client and stores it in a cookie, which means a visitor can switch languages and keep that choice while browsing the site. `components/layout/navbar.tsx` is responsible for the main navigation and the desktop/mobile language dropdown behavior, while `components/layout/footer.tsx` gives the site a consistent closing section with quick links and contact information. I especially liked this separation because it keeps the top-level app easy to reason about while letting shared UI behavior stay in focused component files.

The individual route files each serve a clear purpose. `app/page.tsx` builds the homepage experience with a hero section, feature cards, a mission statement, and summary statistics. `app/curriculums/page.tsx` lays out the curriculum catalog and includes a “how it works” section so the site feels educational rather than purely promotional. `app/blog/page.tsx` creates a featured article area and a grid of posts to make the site feel active and content-rich. `app/workshops/page.tsx` focuses on community engagement by presenting upcoming and past workshops with dates, locations, age ranges, and expectations. `app/projects/page.tsx` is especially important because it creates a filterable project gallery, letting users narrow the list by engineering, science, coding, or robotics. That page is powered by the shared guide data, which helped me avoid repeating descriptions and materials in multiple places.

The project-guide architecture is one of the parts I am most proud of. I used a hybrid approach instead of forcing every guide into the exact same layout. The generic route file `app/projects/[slug]/page.tsx` renders a reusable detail page for guides that only need the standard structure: introduction, explanation, materials, steps, safety, and challenge mode. At the same time, I created dedicated route wrappers such as `app/projects/popsicle-stick-bridge/page.tsx`, `app/projects/lego-robot-builder/page.tsx`, `app/projects/simple-circuit-light/page.tsx`, `app/projects/my-first-python-program/page.tsx`, and `app/projects/baking-soda-volcano/page.tsx`. Those wrappers render richer custom components from the `features/projects/components/` folder. This was a deliberate design choice. I wanted the project to stay scalable through reusable data, but I also wanted standout activities to feel more immersive. Because of that, the custom guide components include more specialized teaching layouts like progress indicators, tabs, accordions, printable resources, diagrams, and build-along sections that would be awkward to force into a one-size-fits-all template.

The `features/projects/components/` folder therefore contains some of the most detailed work in the repository. The bridge guide expands the basic project into a more visual engineering lesson with anatomy terms, printable references, and a build-along format. The LEGO robot guide emphasizes mechanics and robotics thinking. The Python guide turns the lesson into a more code-centered experience. The volcano and circuit guides use interactive presentation patterns that make the instructions feel more like an activity workbook than a plain article. Supporting these components is the trimmed `components/ui/` folder, which now keeps only the reusable primitives the live site actually uses. Those UI building blocks helped me move faster while keeping the more customized pages visually consistent with the rest of the site.

Styling and visual identity are centered in `app/globals.css`, where I defined the Avanza color tokens such as green, teal, purple, orange, and dark navy. I wanted the palette to feel bright and youthful without looking random, so those colors repeat across the hero sections, category tags, buttons, and cards. The `public/` folder stores the supporting assets for that visual identity, including photos from workshops, project images, icons, and the printable bridge template. Using real images was important to me because it makes the site feel connected to actual students and real STEM activities rather than just abstract placeholders.

A few design debates shaped the final result. One question was whether I should make the site fully database-driven. I decided not to, because this version of the project benefits more from clarity and reliability than from backend complexity. Static content in typed files is easier to debug, easier to translate, and easier to present for a class project. Another choice was how to handle localization. I could have implemented route-based internationalization, but I chose a cookie-backed language provider instead because it kept the app simpler while still supporting a multilingual experience across both shared components and project content. I also debated whether every project guide should share one identical structure. In the end, the hybrid model felt like the best compromise between maintainability and creativity.

If I continued developing Avanza STEM, the next improvements would be connecting the workshop buttons to real backend services, adding richer curriculum detail pages instead of overview cards only, and expanding the number of projects supported by the custom lesson layouts. Even in its current form, though, the project shows the full direction I wanted: a community-centered STEM platform that mixes approachable design, multilingual support, reusable architecture, and hands-on educational content in one cohesive web application.

To run the project locally, install dependencies with `npm install` and start the development server with `npm run dev`. Then open the local Next.js URL in the browser to explore the site.

## Newsletter setup

The blog newsletter form posts to `app/api/subscribe`, which validates the email address and sends a notification email through Resend.

1. Copy `.env.example` to `.env.local`.
2. Set `RESEND_API_KEY` to a valid Resend API key.
3. Verify your sending domain in Resend before testing delivery to `liam@avanzastem.org`.
4. Optionally set `NEWSLETTER_FROM_EMAIL` to a sender identity you have configured in Resend. If you leave it blank, the app uses `Avanza STEM <newsletter@avanzastem.org>`.
5. Optionally set `NEWSLETTER_RECIPIENT_EMAIL` if you want signups to go to a different inbox. If you leave it blank, signups are emailed to `liam@avanzastem.org`.

The notification includes the subscriber `email`, `source`, `page`, and `submittedAt`, plus a subject line tailored to newsletter signups.

For production delivery, use a sender address on a domain you have verified in Resend.
